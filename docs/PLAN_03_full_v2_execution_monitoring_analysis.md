# Plan 03: Full v2 Execution, Monitoring, and Result Analysis

## Summary

This plan executes the formal v2 r-out-of-K CIFAR-5 experiment, waits until the full grid completes, and then analyzes the real generated results.

- Select `r` only on `fusion_val`.

Run the full 30-epoch grid with the benchmark-selected configuration B:

```powershell
E:\pytorch_cuda_env\python.exe run_routofk_overlap_experiments.py `
  --data-dir E:\edl_cifar5_fastai `
  --output-root E:\edl_cifar5_runs\routofk_overlap_v2_fusionval `

  --skip-existing
```

Expected output root:

```text
E:\edl_cifar5_runs\routofk_overlap_v2_fusionval
```

Expected condition count:


## Monitoring Policy

Monitoring does not require repeated user approval. After starting the experiment, continue waiting and checking status until one of these terminal states is reached:

- all 30 conditions complete and analysis CSV files are generated;
- the runner exits with an error;
- all Python training processes stop unexpectedly and the output set is incomplete;
- a hard failure appears in logs, such as OOM, NaN, split validation failure, or missing checkpoints.

Use these checks during the run:

```powershell
nvidia-smi --query-gpu=utilization.gpu,memory.used,memory.free --format=csv,noheader,nounits
Get-Process python -ErrorAction SilentlyContinue | Select-Object Id,ProcessName,CPU,WorkingSet,Path
Get-ChildItem -LiteralPath E:\edl_cifar5_runs\routofk_overlap_v2_fusionval -Directory
```

For active or failed conditions, inspect:

```text
condition_stdout.log
condition_stderr.log
manifest.json
splits.json
fusion_val_outputs.npz
test_outputs.npz
```

## Failure and Resume Policy

Do not delete incomplete runs.

If the command is interrupted or a tool timeout occurs but Python processes are still running, keep monitoring the existing processes.

If the runner stops and some conditions are incomplete, rerun the exact same full command. The `--skip-existing` flag will skip conditions that already contain both:

```text
fusion_val_outputs.npz
test_outputs.npz
```

If a condition repeatedly fails:

- read its `condition_stderr.log` and `condition_stdout.log`;
- report the failing condition name, exit code, and relevant error text;
- do not alter seeds, model list, split sizes, or analysis protocol without a new plan.

## Result Analysis Plan

Analyze only real CSV outputs from the completed run.

Main questions:

- How does overlap ratio affect the selected `r`?
- Does `class-skewed-private` select different `r` values than `stratified-balanced`?
- Does the selected `r` improve test accuracy, NLL, Brier score, or ECE compared with single models?
- Is the selected `r` stable across seeds?
- Which classes drive changes in per-class selected `r`?
- Do diversity metrics correlate with selected `r` or test accuracy?
- How large is the gap between formal `fusion_val` selected `r` and diagnostic test-oracle `r`?

Report format:

- concise table of selected `r` frequency by overlap and partition mode;
- aggregate test metrics with mean, standard deviation, and 95% CI;
- comparison against individual model metrics;
- diversity correlation interpretation;
- per-class observations;
- limitations and non-claims;
- concrete improvement directions.

## Improvement Directions to Consider After Results

Do not implement these during the full run. Use them only after the formal result is analyzed.

- Add repeated statistical tests or bootstrap confidence intervals if seed variance is high.
- Add calibration-focused selection criteria when ECE or NLL contradict accuracy.
- Compare r-out-of-K with average, conjunctive, disjunctive, and discount rules under the same fusion-val protocol.
- Add stronger but still lightweight classifiers only as a separate ablation.
- Tune EDL training hyperparameters only after confirming that the current models are undertrained or unstable.
- Add result packaging scripts for publication tables and figures if the CSV analysis is stable.
